### 走过的弯路与经验

#### 1. 没有提前验证 Python 版本兼容性
**问题：** 直接写 `anthropic==0.34.0` 进 requirements，Python 3.14 无法编译 `jiter`（Rust 依赖），安装失败。
**浪费：** 反复排查安装错误，最终改用 requests 直接调 API。
**经验：** 遇到有 Rust/C 扩展的包，先确认用户 Python 版本是否有预编译 wheel，或者直接用 requests 调 HTTP API 规避 SDK 依赖。

#### 2. 没有先测试第三方 API 是否可用
**问题：** Papers With Code API 已并入 Hugging Face，接口返回 HTML 而非 JSON，直接写进代码后才发现。
**浪费：** 写完 fetcher 才发现 API 失效，重写整个模块。
**经验：** 使用第三方 API 前先用 curl 测试一个端点，确认返回 JSON 且结构符合预期，再动手写代码。

#### 3. 没有先探测 API 响应结构就写解析代码
**问题：** HF Daily Papers API 的结构是 `item["paper"]["id"]`，但第一版代码按猜测写成 `item.get("id")`，导致解析失败、抓取结果为 0。
**浪费：** 服务器重启测试才发现问题，再调试修复。
**经验：** 对陌生 API，先单独 curl 一条数据，打印所有字段确认结构，再写解析逻辑。

#### 4. 在 Windows 上用 Linux 进程管理命令
**问题：** 用 `pkill -f "python app.py"` 杀进程，在 Windows 上无效，导致旧进程堆积（同时有 9 个进程监听 5000 端口），新代码无法生效，新路由返回 404。
**浪费：** 反复重启仍无效，最终不得不换端口（5000 → 5001），且排查过程花了大量时间。
**经验：** Windows 环境下进程管理用 `taskkill /F /IM python.exe /T`。每次重启前用 `netstat -ano | grep ":端口"` 确认端口已释放，再启动新进程。

#### 5. 翻译 API 选型没有调研限制
**问题：** 选用 MyMemory 作为免费翻译 API，没有考虑到其单次 500 字符限制和速度慢的问题，用户反馈"特别卡"。
**浪费：** 功能上线后才暴露问题，重写翻译模块。
**经验：** 选 API 前先查清楚：速率限制、字符限制、国内访问速度。学术摘要通常 1500+ 字符，MyMemory 不适合。Google Translate 非官方接口无字符限制且速度快，应该一开始就选它。

#### 6. 数据库迁移函数调用顺序错误
**问题：** 把 `_migrate(conn)` 放在 `executescript` 之前，对还不存在的表执行 `PRAGMA table_info`，逻辑错误。
**经验：** 迁移函数必须在建表语句之后调用，顺序：建表 → 迁移补列。
